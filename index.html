Gentle AI

El framework que hace magia detras de OpenCode.

Que es Gentle AI?

Definicion

Un framework open-source que coordina subagentes especializados para ejecutar el ciclo SDD completo dentro de OpenCode.

Componentes

  • Orchestrator — coordina todo
  • Skills — instrucciones reutilizables
  • MCP Servers — herramientas externas
  • Engram — memoria persistente

Instalacion de Gentle AI

# 1. Clonar el repositorio
git clone https://github.com/gentle-ai/opencode.git

# 2. Instalar dependencias
cd opencode && npm install

# 3. Configurar modelos en opencode.json
{
  "provider": {
    "openrouter": {
      "apiKey": "sk-or-...",
      "models": {
        "gentle-orchestrator": "anthropic/claude-sonnet-4",
        "sdd-apply": "google/gemini-2.5-flash"
      }
    }
  }
}

# 4. Instalar skills
opencode skills install gentle-ai

# 5. Verificar
opencode --version

Protocolo estandar para conectar LLMs con herramientas externas. Permite a los agentes acceder a bases de datos, APIs, archivos, y servicios.

Clean Architecture

Capas, dependencias, y separacion.

Las 3 Capas

D

Domain

Entidades puras, interfaces de repositorios. Sin dependencias externas.

Da

Data

Implementaciones, APIs, bases de datos. Implementa las interfaces de Domain.

P

Presentation

UI, screens, widgets, providers. Consuma Domain.

SOLID

5 principios, 5 letras, 1 objetivo.

S — Single Responsibility

Una clase, una razon para cambiar.

Cada modulo hace una sola cosa. Si cambian dos cosas juntas, estan mal separadas.

// MALO: hace todo
class UserService {
  login() { }
  saveToDb() { }
  sendEmail() { }
}

// BUENO: responsabilidad unica
class AuthService {
  login() { }
}
class UserRepository {
  save() { }
}
class EmailService {
  send() { }
}

O — Open/Closed

Abierto para extension, cerrado para modificacion.

Agrega comportamiento sin tocar el codigo existente. Usa interfaces y composicion.

// MALO: modifcar para agregar
class Discount {
  apply(type, price) {
    if (type == 'vip') return price * 0.8;
    if (type == 'normal') return price * 0.9;
  }
}

// BUENO: extender sin modificar
abstract class Discount {
  apply(price);
}
class VipDiscount implements Discount {
  apply(price) => price * 0.8;
}
class NormalDiscount implements Discount {
  apply(price) => price * 0.9;
}

L — Liskov Substitution

Subtipos deben ser sustituibles por sus tipos base.

Si B es subtipo de A, entonces donde esperas A, debe funcionar con B sin romper nada.

// MALO: rompe el contrato
class Bird {
  fly() { }
}
class Penguin extends Bird {
  fly() { throw Error("No puedo"); }
}

// BUENO: interfaces correctas
abstract class Bird {
  move();
}
class FlyingBird extends Bird {
  move() => this.fly();
}
class SwimmingBird extends Bird {
  move() => this.swim();
}

I — Interface Segregation

Ningun cliente debe depender de metodos que no usa.

Interfaces pequeñas y especificas. No una gigante que haga todo.

// MALO: interfaz gigante
interface Worker {
  work();
  eat();
  sleep();
}

// BUENO: interfaces segregadas
interface Workable {
  work();
}
interface Eatable {
  eat();
}
class Robot implements Workable {
  work() { }
}
class Human implements Workable, Eatable {
  work() { }
  eat() { }
}

D — Dependency Inversion

Depende de abstracciones, no de concreciones.

Los modulos de alto nivel no deben depender de los de bajo nivel. Ambos dependen de interfaces.

// MALO: dependencia concreta
class UserRepository {
  final db = MySqlDatabase(); // atado a MySQL
}

// BUENO: depende de abstraccion
abstract class Database {
  save(String table, Map data);
}
class UserRepository {
  final Database db; // puede ser MySQL, Postgres, Mock
  UserRepository(this.db);
}

Clean Code

Codigo que se lee como prosa.

Principios de Clean Code

Nombres claros

d → daysUntilExpiration
Que hace, no como lo hace

Funciones pequenas

Una funcion = una accion. Max 20 lineas. Sin parametros de 3+.

Comentarios

No expliques QUE, explica POR QUE. Si necesitas explicar QUE, renombra.

Sin magic numbers

if (status == 2) → if (status == StatusActive)

Error handling

Nunca vacios. Usa exceptions tipadas. Log con contexto.

DRY

Don't Repeat Yourself. Pero no prematuramente — esperar a 3 repeticiones.

TDD

Red → Green → Refactor

R

RED

→
G

GREEN

→
Re

REFACTOR

Limpiar sin cambiar comportamiento

TDD en Flutter — Ejemplo

// RED: test que falla
test('should return default greeting', () async {
  final repo = GreetingRepositoryImpl();
  final greeting = await repo.getGreeting();
  expect(greeting.message, 'Hola Mundo');
});

// GREEN: implementar minimo
class GreetingRepositoryImpl implements GreetingRepository {
  @override
  Future getGreeting() async {
    return Greeting.defaultGreeting;
  }
}

// REFACTOR: limpiar
// (ya esta limpio — no hay nada que refactorizar)

TDD no es sobre tests. Es sobre diseno.

Reglas UI/UX Obligatorias

Touch Targets

Minimo 44pt para cualquier elemento interactivo

Contraste

Minimo 4.5:1 para texto, 3:1 para graficos

Dark Mode

Paridad total. Mismo contenido, mismas prioridades visuales

Accessibility

Labels semanticos, reduced-motion, screen readers

Glassmorphism

Blur 10-20px, bordes sutiles, opacidad 0.1-0.25

Grid 8pt

Todos los espacios multiplos de 8: 8, 16, 24, 32, 48, 64
